## 3. 핵심 문제 해결 및 성과

> 이력서에서 강조했던 **트러블 슈팅**을 상세히 적는 칸입니다. (앞서 작성해 드린 Case Study 내용)

### **Trouble 1:** 비동기 처리를 통한 성능 최적화

### 1. Problem (문제 정의)

- **상황:** 사용자 회원가입 시 필수 과정인 이메일 인증 API의 응답 시간이 평균 **2.5초**로 측정됨.
- **원인:** 외부 SMTP 서버와의 통신이 **동기적(Synchronous) 블로킹** 방식으로 처리되어, 메일 발송이 완료될 때까지 사용자가 다음 단계로 진행하지 못하는 병목 현상 발생.

### 2. Action (엔지니어링 의사결정)

- **가설:** "인증 메일 발송은 즉각적인 응답이 필요한 비즈니스 로직과 분리될 수 있다. 이를 비동기화하면 사용자 체감 성능을 대폭 개선할 수 있을 것이다."
- **해결 방안:** **Spring Event**와 **`@Async`**를 활용한 비동기 아키텍처 도입.
  - **Step 1:** 회원가입 로직에서 메일 발송 코드를 분리하여 `ApplicationEvent` 발행.
  - **Step 2:** 별도의 Thread Pool을 구성하여 메일 발송 이벤트를 비동기적으로 처리.
  - **Step 3:** `@EventListener`와 `@Async`를 조합하여 서비스 간 결합도를 낮춤.

### 3. Result (수치화된 성과)

- **성능 개선:** API 응답 시간을 **2.5s에서 0.2s로 92% 단축**.
- **처리량 향상:** 블로킹 타임 제거로 인해 동일 자원 대비 서버 처리량(Throughput) 약 **10배 향상**.
- **사용자 경험:** 회원가입 완료 후 즉시 다음 화면으로 전환되는 매끄러운 UX 제공.

### 4. Deep Dive

> 블로그 글 링크 및 간략한 설명

- **예외 처리:** 비동기 스레드에서 발생하는 예외가 메인 트랜잭션에 영향을 주지 않도록 Error Handler 구현.
- **데이터 일관성:** `TransactionPhase.AFTER_COMMIT`을 활용하여, 회원 정보가 DB에 최종 저장된 후에만 인증 메일이 발송되도록 설계하여 데이터 부정합 방지.
